# The Null Keyboard

In 1984, when Apple shipped the original Macintosh, they made a decision that haunts computing to this day: Command-S to save. The modifier key—a "command" to the machine—became the universal grammar for shortcuts.

But there's an older tradition. On the IBM 3270 terminals of the 1970s, function keys were *dedicated*. F1 was always help. F3 was always exit. No modifier required. The keyboard itself was the interface—each key a direct command, not a chord.

The tension between these philosophies plays out every time someone sets up a new machine. macOS claims Cmd+Shift+3 for screenshots. Apps want their own bindings. Users have muscle memory from other systems. The solution most systems landed on: let users *clear* a binding entirely. Not rebind—just remove. Let the null state be valid.

This is surprisingly rare in software design. Most systems demand a value. A dropdown needs a selection. A field needs input. But the most flexible interfaces treat *absence* as a first-class choice. The keyboard shortcut that doesn't exist is itself a configuration.
